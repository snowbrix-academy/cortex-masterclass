The Problem

This Is How Most Teams Ingest Data

Traditional Approach

  • 100 data sources = 100 COPY INTO scripts
  • Each script is slightly different
  • Change = code deployment
  • New source = new code review
  • Error handling? "It depends on who wrote it"

MDF Approach

  • 100 data sources = 100 config rows
  • One generic procedure handles all
  • Change = update a config row
  • New source = INSERT statement
  • Error handling is consistent everywhere

Key Concepts

Metadata-Driven = Config Over Code

-- Traditional: One script per source (hardcoded) COPY INTO raw_db.sales.orders FROM @s3_stage/sales/orders/ FILE_FORMAT = (TYPE='CSV' SKIP_HEADER=1) PATTERN = '.*orders.*[.]csv' ON_ERROR = 'CONTINUE'; -- MDF: One procedure reads config, builds COPY dynamically CALL SP_GENERIC_INGESTION('DEMO_ORDERS_CSV', FALSE); -- ^ That's it. Config table tells the procedure everything.

The procedure reads the INGESTION_CONFIG table, builds the COPY INTO dynamically, executes it, logs the results, and validates the data. Every time. For every source.

Production-Ready

10 Things That Make This Enterprise-Grade

  • 1. Config-driven — all behavior in metadata
  • 2. Full audit trail — every run logged
  • 3. Error isolation — one failure doesn't cascade
  • 4. Idempotent — safe to re-run anytime
  • 5. Multi-format — CSV, JSON, Parquet, Avro, ORC
  • 6. Multi-client — schema-per-client isolation
  • 7. Schema evolution — handles column drift
  • 8. Cost-optimized — resource monitors + auto-suspend
  • 9. RBAC — least-privilege role hierarchy
  • 10. Observable — 7 monitoring views
